# Instructions

For each bounded context, design its aggregates — the transactional consistency boundaries that protect business invariants. An aggregate is a cluster of domain objects (entities and value objects) treated as a single unit for data changes.

## Step 1: Identify Aggregates and their Roots

For each bounded context, determine which aggregates it contains:
- Each aggregate has exactly one root entity — the single entry point through which all external interactions occur
- The aggregate root controls access to all contained entities and value objects
- Use names consistent with the glossary terms and the aggregates mentioned in the event storming
- Assign each aggregate to exactly one bounded context (using the context names from Phase 3)

## Step 2: Define Contained Elements

For each aggregate, list its internal entities and value objects:
- Entities (element_type: "entity") have a unique identity and lifecycle — their identity persists even when attributes change
- Value Objects (element_type: "value_object") are defined entirely by their attributes — two value objects with the same attributes are interchangeable
- For each element, list its key properties (the most important attributes, not an exhaustive schema)
- Keep aggregates small: prefer value objects over entities where identity is not needed

## Step 3: Define Invariants

Invariants are business rules that the aggregate must enforce at all times. They represent the "protection boundary" of the aggregate.
- Name each invariant concisely (e.g., "Maximum Occupancy", "Positive Balance Required")
- Describe the rule in domain language ("A room cannot have more guests than its capacity")
- Good invariants are verifiable — they can be checked programmatically
- Every aggregate should have at least one invariant; if you cannot identify one, the aggregate boundary may be wrong

## Step 4: Define Commands

Each aggregate handles commands — operations that change its state:
- Name commands in imperative form, consistent with the event storming (e.g., "PlaceOrder", "CancelBooking")
- Describe what the command does in business terms
- List the key input parameters
- For each command, list which domain events it emits on success (use event names from Phase 2)
- For each command, list which invariants are checked (using invariant names from Step 3)

## Step 5: Evaluate Aggregate Size

Assess each aggregate's size as:
- "small": 1-2 entities + a few value objects — ideal for concurrency and simplicity
- "moderate": 3-4 entities — acceptable if the consistency boundary requires it
- "large": 5+ entities or a complex object graph — should be examined for potential splitting

In the size_rationale, explain:
- Why this aggregate groups these specific elements together
- If the aggregate is "large", suggest how it could be split while preserving invariants
- If elements could be extracted into a separate aggregate, note this

## Step 6: Document Design Decisions

In design_decisions, list the key rationale behind your aggregate choices:
- Why you grouped certain entities together
- Why you separated elements that might seem related
- Any trade-offs you made between consistency and performance
- Alternative partitionings you considered and why you rejected them

## Design principles to apply

- **Reference by identity across aggregate boundaries**: aggregates should reference other aggregates by ID, not by holding direct object references
- **One transaction per aggregate**: a single command should modify only one aggregate
- **Eventual consistency between aggregates**: if a business process spans multiple aggregates, use domain events to coordinate
- **Prefer smaller aggregates**: when in doubt, make the aggregate smaller and use domain events for coordination
- **Protect real invariants**: only group elements that must be consistent within a single transaction
