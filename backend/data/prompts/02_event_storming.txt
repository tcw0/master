# Instructions

Perform a systematic Event Storming session. Identify all domain events, the commands that trigger them, the actors who initiate those commands, the aggregates that process them, and any reactive policies. Then organize these elements into coherent business process flows.

## Step 1: Identify Commands

Identify every action that an actor can initiate in the system.
- Name commands in imperative form: "PlaceOrder", "RegisterUser", "ApproveApplication"
- For each command, identify who initiates it (the actor — use role names from the glossary)
- For each command, identify which aggregate handles it (the target_aggregate — use entity names from the glossary)

## Step 2: Identify Domain Events

For each command, determine what domain event(s) it produces when successfully executed.
- Name events in past tense: "OrderPlaced", "UserRegistered", "ApplicationApproved"
- Every event must be traceable to exactly one triggering command
- Every event must be emitted by exactly one aggregate
- Use names consistent with the glossary terms

## Step 3: Identify Policies

Policies are reactive business rules that listen for events and may trigger follow-up commands.
- Name policies descriptively: "Send Confirmation Email Policy", "Auto-Cancel Expired Reservations"
- Each policy is triggered by exactly one event
- If a policy initiates a follow-up action, specify the resulting_command; otherwise set it to null
- Not every event triggers a policy — many events are terminal

## Step 4: Organize into Flows

Group related events into named business process flows (e.g., "Order Fulfillment", "User Onboarding", "Payment Processing"). Each flow represents an end-to-end business process.

For each flow, create an ordered sequence of steps. Each step represents one sticky-note group in a classic Event Storming board and must specify:
- actor: who initiates the action
- command: what action is performed
- aggregate: which aggregate handles it
- event: what domain event is produced
- policy: if a reactive policy is triggered, name it (null otherwise)
- next_command: if the policy triggers a follow-up command, name it (null otherwise)

Steps within a flow should follow the temporal/causal order of the business process.

## Step 5: Flag ambiguities

List any areas where:
- The event flow is unclear or could be interpreted multiple ways
- The requirements do not specify who initiates an action or what happens next
- There are potential race conditions or parallel processes that need clarification

## Naming conventions

- Commands: imperative verb + noun (e.g., "PlaceOrder", "CancelBooking")
- Events: past tense verb + noun (e.g., "OrderPlaced", "BookingCancelled")
- Aggregates: singular nouns matching glossary entity terms (e.g., "Order", "Booking")
- Actors: role names matching glossary role terms (e.g., "Customer", "Administrator")
- Policies: descriptive name of the business rule (e.g., "Reservation Expiration Policy")

## Consistency requirements

- Every command name used in flows must appear in the commands list
- Every event name used in flows must appear in the domain_events list
- Every policy name used in flows must appear in the policies list
- Aggregate and actor names should match terms from the glossary
