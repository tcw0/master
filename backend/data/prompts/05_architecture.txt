# Instructions

For each bounded context, design a hexagonal (ports and adapters) architecture that separates domain logic from technical infrastructure. Also define anti-corruption layers between contexts and published interfaces for external communication.

## Step 1: Map Each Bounded Context to a Hexagonal Architecture

For each bounded context, define elements in all four layers. Every element must have a name, type, and description of its responsibility.

### Domain Layer (innermost — pure business logic, no dependencies on infrastructure)

Identify elements of these types:
- "entity": domain entities from the aggregate design (the aggregate roots and contained entities)
- "value_object": value objects from the aggregate design
- "domain_service": operations that do not naturally belong to a single entity — they coordinate logic across aggregates or encapsulate domain rules
- "repository_interface": abstractions (interfaces/ports) for accessing aggregates — define the contract, not the implementation
- "domain_event": events published by aggregates (from Phase 2)

### Application Layer (orchestrates use cases, no domain logic)

Identify elements of these types:
- "application_service": services that implement use cases by coordinating domain objects and infrastructure — one per major use case
- "dto": data transfer objects for input/output of application services
- "command": command objects representing write intentions (from Phase 2 commands)
- "query": query objects representing read requests
- "command_handler": handlers that process commands and delegate to domain objects
- "query_handler": handlers that process queries and return DTOs

### Infrastructure Layer (technical adapters implementing ports)

Identify elements of these types:
- "repository_implementation": concrete implementations of domain layer repository interfaces (e.g., PostgreSQL, MongoDB)
- "external_adapter": adapters for external systems (e.g., payment gateway, email service, third-party APIs)
- "event_publisher": infrastructure for publishing domain events (e.g., message broker adapter, event bus)
- "persistence_config": database schemas, ORM mappings, connection configuration
- "messaging": message queue consumers, event listeners, integration adapters

### Presentation Layer (outermost — external interface adapters)

Identify elements of these types:
- "rest_controller": REST API endpoints exposing application services
- "graphql_resolver": GraphQL resolvers (if applicable based on the domain)
- "api_dto": request/response DTOs specific to the API format
- "middleware": cross-cutting concerns such as authentication, logging, rate limiting

## Step 2: Design Anti-Corruption Layers

For each pair of bounded contexts that requires model translation (identified by "anti_corruption_layer" or "conformist" relationships in Phase 3), define an ACL:
- owning_context: the context that owns and maintains the ACL
- foreign_context: the context whose model is being translated from
- translation_description: what concepts are translated and how (e.g., "Translates external PaymentProvider events into internal Payment domain events")
- translated_elements: the specific domain elements that pass through this ACL

Even if no "anti_corruption_layer" relationship was explicitly identified in Phase 3, consider whether any upstream/downstream relationships would benefit from translation to protect the downstream model.

## Step 3: Define Published Interfaces

For each bounded context, define the interfaces it exposes for external consumption:
- "rest_api": RESTful HTTP APIs
- "graphql_api": GraphQL endpoints
- "domain_events": events published for other contexts to subscribe to
- "shared_kernel": shared model elements co-owned with another context

Each interface should list its exposed_operations — the specific endpoints, event types, or shared model elements.

## Step 4: Document Technical Patterns

For each technical pattern applied, explain:
- pattern_name: the name of the pattern (e.g., "Repository", "CQRS", "Specification", "Domain Events", "Saga", "Unit of Work")
- applied_in_context: which bounded context uses this pattern
- justification: why this pattern is appropriate — how it supports the domain model and what problem it solves

## Architecture principles to apply

- **Dependency rule**: dependencies point inward — domain layer has no dependencies on application, infrastructure, or presentation layers
- **Domain purity**: the domain layer must be free of framework dependencies, database concerns, and I/O operations
- **One context, one deployable**: each bounded context should be independently deployable
- **Explicit interfaces**: all communication between contexts goes through published interfaces, never direct database access
- **Test-friendly**: the hexagonal structure enables testing domain logic in isolation from infrastructure
