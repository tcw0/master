# Instructions

Analyze the domain model to identify bounded contexts — cohesive areas of the domain where a specific ubiquitous language applies consistently. Then map the relationships between these contexts.

## Step 1: Identify Bounded Contexts

Look for natural boundaries in the domain by examining:
- **Linguistic boundaries**: terms that have different meanings in different parts of the domain suggest separate contexts
- **Process boundaries**: aggregates and events that participate in the same business processes often belong to the same context
- **Organizational boundaries**: different teams or departments often correspond to different contexts
- **Data ownership**: entities that are owned and primarily managed by the same part of the business belong together

For each bounded context, provide:
- A clear, descriptive name
- Its core purpose: what business capability does it own? (one or two sentences)
- Strategic classification (domain_type):
  - "core": key competitive differentiator, high business value, requires deep domain expertise
  - "supporting": necessary but not a differentiator, supports core contexts
  - "generic": commodity capability that could be outsourced or replaced by an off-the-shelf solution (e.g., authentication, email, notifications)
- Key aggregates: which aggregates from the event storming belong here
- Ubiquitous language terms: which glossary terms are specific to this context

## Step 2: Define Context Relationships (Context Map)

For every pair of contexts that interact, define the relationship using one of these DDD integration patterns:
- "upstream_downstream": one context provides data/services that another depends on
- "shared_kernel": two contexts share a common sub-model that both teams co-own
- "customer_supplier": the downstream context can negotiate requirements with the upstream
- "conformist": the downstream context accepts whatever the upstream provides
- "anti_corruption_layer": the downstream context translates foreign concepts to protect its own model
- "published_language": the upstream context publishes a well-defined shared language (e.g., an API schema)
- "open_host_service": the upstream context provides an open protocol for multiple consumers
- "partnership": two contexts evolve together with mutual coordination
- "separate_ways": two contexts have no integration and operate independently

For each relationship:
- source_context is the upstream / provider context
- target_context is the downstream / consumer context
- Provide a description explaining how the contexts interact

## Step 3: Identify Term Overlaps

List any terms from the glossary that have different meanings in different bounded contexts. For each overlapping term:
- Name the term
- For each context where it appears, explain what it means in that specific context

These term overlaps are critical signals for correct context boundaries and anti-corruption layer design.

## Design principles to apply

- **Prefer smaller, focused contexts** over large monolithic ones
- **Each aggregate should belong to exactly one bounded context** — if an aggregate seems to span contexts, it may need to be split
- **A bounded context should have a single, clear purpose** — if you struggle to state it in one sentence, the context may be too broad
- **Avoid creating a "generic/shared" context** as a catch-all — each context should own specific behavior
- **Question any context with more than 5-6 aggregates** — consider whether it should be split
